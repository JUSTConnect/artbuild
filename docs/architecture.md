# Architecture

## Цель архитектуры

Архитектура проекта должна быть расширяемой и не превращаться в монолит. В игре будет много систем: строительство, жители, энергия, прогрессия, генерация вариантов, визуальная сборка башни, UI, сохранения, share-видео. Если всё смешать в один главный класс, проект быстро станет сложно развивать.

Главный принцип:

```text
Данные отдельно. Логика отдельно. Визуал отдельно. UI отдельно. Сохранения отдельно.
```

## Слои проекта

Рекомендуемая логическая структура:

```text
Artbuild
├─ App
├─ Presentation
├─ Game
├─ Domain
├─ Content
├─ Infrastructure
└─ Tools
```

### App

Точка входа приложения. Отвечает за запуск игры, инициализацию сервисов, загрузку сцены и связывание модулей.

Не должна содержать правила строительства, расчёты энергии или прогрессию.

### Presentation

Визуальный слой:

- отрисовка башни;
- анимации появления домиков;
- подсветка доступных мест;
- UI выбора из трёх вариантов;
- всплывающие подсказки;
- визуальное состояние энергии и жителей.

Presentation получает готовое состояние от Game/Domain и показывает его игроку.

### Game

Application-слой, который связывает действия игрока с доменной логикой.

Примеры сервисов:

- `BuildCommandService` — обработка команды «поставить домик»;
- `ChoiceService` — генерация трёх вариантов;
- `ProgressionService` — открытие новых уровней и типов зданий;
- `SessionService` — управление текущей игровой сессией;
- `ShareReplayService` — подготовка данных для ролика роста башни.

### Domain

Чистая игровая логика без зависимости от UI и конкретного движка.

Сюда входят:

- модель башни;
- правила размещения;
- расчёт энергии;
- расчёт жителей;
- расчёт красоты, технологии и комфорта;
- условия открытия новых зданий;
- проверка доступных слотов.

Domain должен быть максимально тестируемым.

### Content

Данные и конфиги:

- описания зданий;
- параметры домиков;
- редкость;
- размеры;
- типы соединений;
- визуальные ID ассетов;
- правила появления;
- требования для разблокировки.

Контент лучше хранить в data-driven формате, чтобы добавление нового домика не требовало переписывать код.

### Infrastructure

Технические сервисы:

- сохранения;
- загрузка ассетов;
- аналитика;
- локализация;
- настройки;
- запись видео или replay-данных;
- интеграции с мобильной платформой.

### Tools

Внутренние инструменты разработки:

- редактор домиков;
- валидатор точек стыковки;
- генератор превью;
- тестовая песочница башни;
- инструмент проверки баланса.

## Основные доменные модели

### TowerState

Хранит текущее состояние башни:

```text
id
placedBlocks
availableSlots
height
population
capacity
energyProduced
energyRequired
beauty
technology
comfort
unlockedBuildingTags
history
```

### BuildingDefinition

Описание типа здания из конфига:

```text
id
name
category
size
allowedConnections
functionOptions
baseStats
unlockRequirements
visualPrefabId
rarity
placementRules
```

### PlacedBuilding

Конкретно размещённый домик:

```text
instanceId
definitionId
position
rotationOrSide
selectedFunction
isPowered
residentCount
connectedSlots
```

### PlacementSlot

Доступная точка размещения:

```text
id
position
connectionType
supportedSize
ownerBuildingId
isOccupied
```

## Поток команды строительства

```text
UI: игрок выбирает домик
↓
BuildCommandService получает команду
↓
PlacementService проверяет, можно ли поставить домик
↓
TowerState обновляется
↓
EnergySystem пересчитывает питание
↓
ResidentSystem пересчитывает жителей
↓
BalanceSystem пересчитывает красоту, технологию и комфорт
↓
ProgressionSystem открывает новые возможности
↓
Presentation анимирует появление домика
```

## Почему так не будет монолита

- UI не знает формулы энергии.
- Энергия не знает, как отрисовывается ветряк.
- Домики описываются конфигами, а не хардкодятся в коде.
- Система прогрессии не создаёт визуальные объекты напрямую.
- Сохранения сохраняют состояние, но не управляют правилами игры.
- Добавление нового здания должно требовать новый конфиг и ассет, а не переписывание центральной логики.

## Data-driven подход

Пример конфигурации здания:

```json
{
  "id": "library_small_01",
  "name": "Small Library",
  "category": "Knowledge",
  "size": { "w": 1, "h": 1 },
  "allowedConnections": ["Top"],
  "functionOptions": ["Knowledge"],
  "baseStats": {
    "beauty": 1,
    "technology": 2,
    "comfort": 1,
    "energyRequired": 1,
    "residentCapacity": 0
  },
  "unlockRequirements": {
    "minTowerLevel": 2
  },
  "visualPrefabId": "building_library_small_01",
  "rarity": "common"
}
```

## Тестируемость

Минимальные тесты для домена:

- нельзя поставить домик в занятый слот;
- нельзя поставить большой домик на маленькую опору;
- ветряк питает дома в радиусе;
- непитаемый дом не заселяется;
- высокая технология открывает технологичные здания;
- низкий комфорт замедляет рост населения;
- выбор из трёх вариантов учитывает дефицит энергии или жилья.
