# Project Rules

Этот файл фиксирует обязательные правила разработки проекта **Artbuild**. Его нужно читать перед изменением архитектуры, добавлением систем, написанием gameplay-кода и созданием новых ассетов/ресурсов.

## 1. Движок и целевая версия

Проект разрабатывается на **Godot 4.6**.

Все технические решения, сцены, скрипты, ресурсы и архитектурные соглашения должны исходить из возможностей и ограничений Godot 4.6.

Если используется функциональность, которая отличается между версиями Godot, нужно явно проверять совместимость именно с Godot 4.6 и не опираться на поведение старых версий движка.

## 2. Не создавать монолит

Запрещено собирать всю игровую логику в одном большом скрипте, одной сцене, одном singleton/autoload или одном менеджере.

Нельзя создавать классы формата:

- `GameManager`, который знает обо всём;
- `TowerManager`, который одновременно отвечает за UI, размещение, энергию, жителей, сохранения и визуал;
- `MainScene`, внутри которой находится вся логика игры;
- `Global`, в котором лежит состояние всех систем.

Каждая система должна иметь одну понятную ответственность.

Пример разделения:

- система башни отвечает за структуру башни;
- система размещения отвечает за проверку доступных точек и установку блоков;
- система энергии отвечает за производство, покрытие и потребление энергии;
- система жителей отвечает за вместимость, заселение и удовлетворённость;
- система прогрессии отвечает за открытие новых блоков и правил выбора;
- UI только отображает состояние и отправляет команды, но не хранит правила игры.

## 3. Ограничение размера файлов

Один файл не должен превышать **400 строк**.

Это правило относится к:

- `.gd` скриптам;
- markdown-документации;
- JSON/ресурсным конфигам, если они становятся нечитаемыми;
- вспомогательным editor/tool-скриптам.

Если файл приближается к лимиту, его нужно разделить по смыслу до того, как он станет сложным для поддержки.

Разделение должно быть архитектурным, а не механическим. Нельзя просто перенести случайные функции в `utils.gd`. Нужно выделять новую ответственность: отдельную систему, компонент, ресурс, сервис, фабрику или валидатор.

## 4. Один источник истины для каждой системы

Для каждой отдельной системы должен существовать один главный источник истины.

Это означает, что важные данные не должны дублироваться в разных местах.

Примеры источников истины:

- список типов зданий хранится в `BuildingCatalog`, а не размазан по UI, сценам и скриптам;
- правила энергии хранятся в `EnergySystem`, а не внутри каждого домика;
- требования к прогрессии хранятся в `ProgressionSystem` или ресурсах прогрессии;
- правила размещения хранятся в `PlacementSystem`, а не в кнопках UI;
- состояние башни хранится в модели башни, а визуальная сцена только отражает это состояние.

Если система должна использовать данные другой системы, она обращается к её публичному API или получает снапшот/событие, а не копирует данные к себе.

## 5. Не плодить хардкод

Запрещено зашивать игровые значения напрямую в случайные скрипты.

Нельзя делать так:

```gdscript
if building_type == "library":
    beauty += 3
    technology += 1
    residents += 2
```

Правильный подход: данные здания должны быть описаны в ресурсе, каталоге или конфиге, а система должна читать эти данные.

Например:

- `BuildingDefinition` описывает тип здания;
- `BuildingCatalog` хранит доступные определения;
- `ProgressionSystem` выбирает доступные здания;
- `PlacementSystem` проверяет, можно ли поставить выбранный блок;
- `TowerState` хранит результат размещения.

Игровой баланс должен быть data-driven. Числа, веса, радиусы энергии, вместимость жителей, прирост красоты и технологии должны быть в данных, а не внутри UI или визуальной сцены.

## 6. Системы важнее случайных решений

Любая новая механика должна добавляться через систему или расширение существующей системы.

Перед добавлением кода нужно определить:

1. К какой системе относится новая логика?
2. Где находится источник истины для этой логики?
3. Какие данные должны быть ресурсами или конфигами?
4. Какие события или команды нужны для связи с другими системами?
5. Как это можно будет расширить позже без переписывания старого кода?

Если на эти вопросы нет ответа, код добавлять рано.

## 7. Разделение данных, логики и визуала

В проекте нужно разделять три слоя:

### Данные

Ресурсы, конфиги, определения зданий, параметры энергии, требования прогрессии, веса выбора и визуальные метаданные.

### Логика

Системы, которые принимают решения: можно ли поставить домик, хватает ли энергии, сколько жителей заселится, какие здания доступны дальше.

### Визуал

Godot-сцены, анимации, спрайты, эффекты, подсветки, UI и декоративные элементы.

Визуал не должен принимать ключевые gameplay-решения. Он должен получать состояние от логики и отображать его.

## 8. Рекомендуемая архитектура Godot

Базовое направление структуры проекта:

```text
res://
  scenes/
    game/
    ui/
    buildings/
  scripts/
    core/
    systems/
    data/
    presentation/
    ui/
  resources/
    buildings/
    progression/
    balance/
  assets/
    sprites/
    audio/
    fonts/
  docs/
```

Рекомендуемые сущности:

- `BuildingDefinition` — ресурс с описанием типа здания;
- `BuildingInstance` — размещённый экземпляр здания в башне;
- `TowerState` — модель текущей башни;
- `PlacementSystem` — правила размещения;
- `EnergySystem` — производство, радиус и потребление энергии;
- `PopulationSystem` — жители, вместимость и удовлетворённость;
- `ProgressionSystem` — открытие новых возможностей;
- `BuildingCatalog` — единый каталог доступных зданий;
- `TowerPresenter` — визуальное отображение состояния башни;
- `GameFlowController` — координация шага игры без хранения всей логики в себе.

## 9. Autoload/singleton использовать осторожно

Autoload в Godot не должен становиться свалкой глобального состояния.

Допустимые случаи:

- сервис сохранения;
- сервис загрузки статических каталогов;
- аудио-сервис;
- сервис настроек;
- маршрутизация между сценами.

Недопустимые случаи:

- хранить всю башню в глобальном объекте без модели;
- держать все правила игры в одном singleton;
- обращаться из любого скрипта к глобальному объекту ради обхода нормальной архитектуры.

## 10. UI не содержит игровые правила

UI может:

- показывать доступные домики;
- показывать энергию, жителей, красоту, технологию и удовлетворённость;
- отправлять команду выбора или размещения;
- отображать ошибки размещения.

UI не должен:

- рассчитывать энергию;
- решать, какие здания открыть;
- хранить реальные параметры здания;
- менять состояние башни напрямую в обход систем.

## 11. События и команды

Связь между системами должна быть явной.

Рекомендуемый поток:

```text
UI → команда → система → изменение модели → событие → визуальное обновление
```

Пример:

```text
Player selects building
→ PlacementSystem validates placement
→ TowerState adds BuildingInstance
→ EnergySystem recalculates coverage
→ PopulationSystem updates residents
→ TowerPresenter redraws changed parts
```

## 12. Документация обязательна для новых систем

Каждая новая крупная система должна иметь короткое описание:

- за что отвечает система;
- какие данные использует;
- какие события принимает и отдаёт;
- где находится источник истины;
- какие ограничения есть сейчас;
- как её можно расширять позже.

Документация может быть в `docs/` или в начале соответствующего файла, если описание короткое.

## 13. Правило спокойной игры

Любая механика должна поддерживать основную идею Artbuild: спокойное, красивое, творческое строительство башни.

Не добавлять механику только потому, что она делает игру сложнее. Сложность должна быть мягкой и понятной: баланс энергии, жителей, красоты, технологии и выбора новых домиков.

Игрок должен чувствовать, что он выращивает красивый вертикальный город, а не борется с таблицами и наказаниями.

## 14. Проверочный список перед добавлением кода

Перед созданием нового кода нужно проверить:

- не превращается ли файл в монолит;
- не превышает ли файл 400 строк;
- есть ли один источник истины для данных;
- нет ли игровых чисел, зашитых в UI или случайные скрипты;
- отделена ли логика от визуала;
- можно ли будет расширить систему без переписывания половины проекта;
- соответствует ли решение Godot 4.6;
- поддерживает ли механика спокойный визуальный характер игры.

## 15. Главный принцип

Сначала система, потом реализация.

Нельзя добавлять случайный код ради быстрого результата, если из-за этого проект становится сложнее расширять. Artbuild должен расти так же, как башня внутри игры: маленькими понятными блоками, которые красиво стыкуются друг с другом.
